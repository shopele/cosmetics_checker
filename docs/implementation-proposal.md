## 1. 優先度 ★★★ 高（実務精度に直結）

### 1.1 判定理由の詳細化と代替案提示

**市場競合**: 薬機法チェック LLMアプリ、TRUSQUETTA AD  
**理由**: 「NG」と判定されても「なぜ？」「どう直すか？」が不明だと実用性が低い

**実装内容**:
```
現在:
  ✗ 記載なし (項目名)

改良後:
  ✗ 記載なし (全成分表示)
    理由: 法定成分の一部が欠落しています
    該当法: 薬機法第61条第2項、通知第27-1
    改善案: 以下の成分が必要です
      - ××成分（含有量%）
      - ○○成分（含有量%）
```

**実装方法**:
- `js/rules.js` のルール定義に `reason`（理由）と `suggestion`（代替案）フィールドを追加
- Claude API の抽出テキスト + ルール照合で、該当理由と改善案を提示
- UI に折りたたみ式で詳細表示

**影響範囲**: `api/check.js`, `js/rules.js`, `js/app.js` の結果表示ロジック

---

### 1.2 NG表現データベースの充実化と業界監修ルール

**市場競合**: PROOFREADER（薬事申請データ照合）、TRUSQUETTA AD（弁護士監修）  
**理由**: 「効能効果標榜」「医学用語の不正使用」などの違反表現は項目チェックでは捕捉できない

**実装内容**:
- NG表現リスト（成分名・医学用語・宣伝表現など）を拡大・カテゴリ分類
- 例:
  ```json
  {
    "category": "禁止表現_効能効果",
    "pattern": ["治す", "治療", "医療", "セラピー"],
    "level": "critical",
    "法例": "薬機法第66条",
    "suggestion": "「整える」「ケアする」に言い換え"
  }
  ```
- Claude API に検査ルール をシステムプロンプトで注入
- 抽出テキストにマッチすれば「NG表現検出」として結果に表示

**実装方法**:
- `js/rules.js` に `NG_EXPRESSIONS` リストを追加
- `api/check.js` で最初に簡易スキャン、Claude API 呼び出し前に前処理
- 結果に「NG表現検出」セクションを追加

**影響範囲**: `js/rules.js`, `api/check.js`, UI コンポーネント

---

### 1.3 複数ファイル形式対応（PDF、テキスト）

**市場競合**: PROOFREADER（PDF）、フォルトファインダープロ（PDF）、TooAI SUITE（複数形式）  
**理由**: 実務では版下 PDF や Excel 原稿、テキスト文書での入力が多い

**実装内容**:
```
現状: 画像ファイルのみ（JPEG, PNG, WebP）

追加対応:
  - PDF（pdfjs-dist ライブラリで自動テキスト抽出）
  - Excel / CSV（テキスト形式への変換）
  - テキスト直接入力（成分表・表示原稿のコピペ）
```

**実装方法**:
- `input[type=file]` の `accept` を拡張
- PDF は `pdfjs-dist` で自動抽出
- Excel/CSV は `papaparse` で解析
- テキスト入力用の別タブを UI に追加
- `api/check.js` で入力形式を統一してから Claude API に送信

**実装難度**: 中程度  
**影響範囲**: UI エリア [B]（ファイル選択拡張）、`api/check.js`, `js/app.js`

## 2. 優先度 ★★ 中（運用効率化）

### 2.1 成分表記の自動検証（成分リスト照合）

**市場競合**: PROOFREADER（薬事申請データベース照合）  
**理由**: 「全成分表示に抜けがないか」を手作業で確認するのは負担が大きい

**実装内容**:
- 公開されている「化粧品成分表示の基準に関する通知」の全成分リストを JSON 化
- パッケージから抽出した成分を、標準リストと照合
- 未記載・順序異常・表記ゆれを検出

**実装方法**:
- `js/ingredients-db.js` に全成分リスト（ISO命名法、日本語名、INCI など）を定義
- `api/check.js` で成分抽出後、DB と照合
- 結果に「成分照合」セクション追加
- localStorage にコーヒー企業の成分リストをキャッシュ（オフライン対応）

**実装難度**: 中程度（データ収集が手作業）  
**影響範囲**: 新ファイル `js/ingredients-db.js`, `api/check.js`, UI コンポーネント

---

### 2.2 チェック結果の詳細レポート生成

**市場競合**: 有料システム全般（監査用エビデンス）  
**理由**: 企業コンプライアンス部門では「誰が、いつ、何をチェックしたか」の記録が必須

**実装内容**:
```
現状: 履歴は JSON で保存、CSV エクスポートのみ

追加:
  - チェック実行時の「メタデータ」を記録
    (チェック者、チェック日時、モデル、規格バージョン など)
  - HTML レポート生成（社内共有・監査用）
  - デジタル署名・タイムスタンプ機能
```

**実装方法**:
- 履歴に `checker_name`, `checker_email`, `rule_version`, `model_used` フィールドを追加
- HTML レポート生成機能を `js/report.js` に追加
- 署名用に crypto.subtle API を活用

**実装難度**: 低〜中程度  
**影響範囲**: `js/app.js`（履歴保存ロジック）、新ファイル `js/report.js`

---

### 2.3 定期チェック・スケジュール監視

**市場競合**: LOGRIZa（定期パトロール）  
**理由**: 新商品の定期的な表示チェックや、更新検知が必要な運用がある

**実装内容**:
```
- パッケージ画像の URL を登録
- 指定時間ごと（日次・週次など）にチェック実行
- 前回との差分を検出
- 変更があれば Slack 通知
```

**実装方法**:
- サーバー側に `cron` ジョブ実装（Node.js `node-cron` パッケージ）
- 監視対象URLのデータベース管理
- Slack Webhook 統合
- クライアント UI で監視設定・管理

**実装難度**: 高（バックエンド実装が必須）  
**影響範囲**: `server.js`, 新エンドポイント `api/schedule.js`, UI コンポーネント

## 3. 優先度 ★ 低（差別化・拡張性）

### 3.1 外部連携・API 提供

**市場競合**: 有料システムの一部は API 提供  
**理由**: 企業のマーケティング自動化、CMS 連携が進んでいる

**実装内容**:
- `/api/check` を公開 API として規格化
- API キー認証、レート制限を強化
- Webhook 通知（チェック結果を外部システムに POST）
- Slack Integration App 化

**実装難度**: 高  
**影響範囲**: `api/check.js`, 認証ロジック新設, `server.js`

---

### 3.2 チームコラボレーション機能

**市場競合**: review-it! for Package（多人数回覧）  
**理由**: 企業では「チェッカー → 薬事責任者 → 承認者」のワークフローが一般的

**実装内容**:
```
- 複数ユーザーのアカウント管理
- 結果の共有・コメント機能
- 承認ワークフロー（Draft → Review → Approved）
- 権限管理（View / Edit / Approve）
```

**実装難度**: 非常に高（認証・DB 設計が必須）  
**影響範囲**: 全体的な大規模リファクタリング必要

---

### 3.3 複数モデル自動比較・精度検証

**理由**: Claude Opus/Sonnet/Haiku の出力結果の差を自動で比較し、信頼度を向上させる

**実装内容**:
```
- チェック実行時に複数モデルで並列実行
- 結果を比較（多数決 or スコアリング）
- 「信頼度」パーセンテージを表示
```

**実装難度**: 中程度  
**API コスト**: 3倍の消費  
**影響範囲**: `api/check.js`, 結果表示ロジック
